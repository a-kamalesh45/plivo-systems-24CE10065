Hybrid Reliable Transport Protocol

Architecture Summary • Systems Assignment

1. Overview

The objective of this assignment was to design a lightweight transport protocol capable of reliably delivering real-time media across an unreliable UDP network while satisfying two constraints imposed by the grading harness:

  • Deadline misses must remain below 1%.
  • Total bandwidth overhead must remain below 2×.

The supplied relay intentionally introduces packet loss, packet duplication, variable network delay and packet reordering. The baseline implementation forwarded every frame exactly once, meaning every dropped or excessively delayed packet permanently contributed to the deadline miss rate.

The final solution therefore combines multiple complementary reliability mechanisms rather than relying on a single technique. Forward Error Correction (FEC) provides immediate recovery for isolated packet loss, receiver-driven retransmissions recover packets that parity cannot restore, while a receiver-side jitter buffer absorbs network delay variation before playback.

Rather than choosing between proactive redundancy or reactive recovery, the protocol combines both approaches. Most packet losses are corrected immediately using parity, while retransmissions are reserved only for losses that remain unrecoverable after FEC processing.

The sender receives original media frames directly from the harness, encapsulates them inside a custom transport protocol and transmits them towards the relay. The relay represents the unreliable network and is responsible for simulating realistic transport impairments. The receiver reconstructs missing packets whenever possible, buffers incoming packets to compensate for network jitter, performs deadline-aware playback scheduling and finally forwards reconstructed media frames back to the harness player.

3. Protocol Design

Instead of transmitting raw harness packets directly across the unreliable network, every outgoing packet is wrapped inside a small transport header. The protocol distinguishes different traffic classes so that media, parity and feedback packets can be processed independently without ambiguity.

Transport Header

struct TransportHeader
{
    PacketType type;

    uint32_t sequence;

    uint32_t block_base;

    uint64_t timestamp;
};
Field Description
Packet Type Identifies MEDIA, FEC, ACK or NACK packets.
Sequence Number Unique identifier for every media frame.
Block Base Identifies the parity block or retransmission target.
Timestamp Generation timestamp used for round-trip time estimation.

Separating packet type from packet sequence significantly simplifies receiver logic compared to embedding flags directly inside the sequence number. The protocol remains compact while being considerably easier to extend with future control messages.

4. Overall Protocol Workflow

Harness ↓ Sender ↓ Generate Media Packet ↓ Transmit Media ↓ Generate XOR Parity ↓ Transmit Parity ↓ Hostile Relay ↓ Receiver ↓ Store Packet ↓ Attempt FEC Recovery ↓ Generate ACK / NACK ↓ Jitter Buffer ↓ Deadline Scheduler ↓ Harness Player

Every media frame therefore has two independent opportunities to survive network impairment. If parity alone is sufficient, recovery happens immediately without any additional network traffic. If parity cannot recover the frame, the receiver requests a selective retransmission using a NACK packet. ACK packets simultaneously allow the sender to reclaim obsolete state, keeping memory usage bounded throughout execution.

5. Sender Architecture

The sender is responsible for converting the raw media stream generated by the harness into a resilient transport stream capable of surviving packet loss and moderate network jitter. Rather than transmitting packets blindly, the sender maintains a bounded history of recently transmitted media frames. This history enables selective retransmission whenever the receiver detects an unrecoverable packet loss.

Harness Frame ↓ Read Sequence Number ↓ Store in Sliding History ↓ Build MEDIA Packet ↓ Transmit ↓ Generate XOR Parity ↓ Transmit FEC Packet ↓ Wait for ACK / NACK ↓ Garbage Collection

Sliding History

Every transmitted media frame is temporarily stored inside a sender-side history indexed by sequence number. This history serves two purposes.

  • Selective retransmission after a receiver NACK.
  • Memory-efficient cleanup after ACK reception.

Unlike an unbounded packet cache, acknowledged packets are periodically removed from memory. This guarantees constant memory usage even during long executions.

Forward Error Correction

For every pair of media packets, the sender generates one XOR parity packet. 


Packet A

Packet B

↓

Parity = A XOR B

If exactly one packet from the pair is lost during transmission, the receiver can reconstruct the missing payload using the surviving media packet together with the parity packet. 


Recovered

=

Parity XOR Received Packet

This recovery requires no interaction with the sender and therefore incurs no additional network latency.

Selective Retransmission

Whenever the receiver determines that parity recovery is impossible, it transmits a NACK containing the missing sequence number. The sender consults its sliding history and retransmits only the requested frame rather than repeating the entire stream. Selective retransmission minimizes bandwidth while still recovering packets that exceed the capability of XOR parity.

6. Receiver Architecture

The receiver performs substantially more work than the sender. Its responsibilities include packet reordering, parity reconstruction, deadline-aware playback scheduling, ACK generation, NACK generation, round-trip telemetry and memory management.

Receive Packet ↓ Packet Classification ↓ MEDIA ? ↓ Store ↓ Attempt Recovery ↓ Generate ACK ↓ Generate NACK (if needed) ↓ Jitter Buffer ↓ Deadline Scheduler ↓ Player

Packet Classification

Incoming packets are immediately classified according to the transport header.

Packet Type Purpose
MEDIA Original media frame.
FEC XOR parity block.
ACK Confirms successful reception.
NACK Requests retransmission.

Separating packet processing by packet type keeps the receiver logic simple and avoids ambiguity during recovery.

7. Jitter Buffer

Network delay varies significantly between packets. Forwarding every packet immediately would cause many frames to miss their playback deadline simply because they arrived slightly later than neighboring frames. To mitigate this problem, the receiver maintains a small jitter buffer. Packets are temporarily stored according to sequence number before playback.

Packet Arrives ↓ Store in Buffer ↓ Wait Until Playback Time ↓ Packet Available? ↓ YES → Play ↓ NO ↓ Attempt Recovery ↓ Still Missing? ↓ Skip

The playout scheduler continuously compares the current wall-clock time against the expected playback time derived from 


T0

+

delay_ms

+

sequence × frame_period

If a packet cannot be recovered before its deadline, playback advances to the next sequence rather than delaying the remainder of the stream. This prevents a single missing packet from causing cascading deadline misses.

8. ACK / NACK Feedback Protocol

Although Forward Error Correction successfully reconstructs most isolated packet losses, parity alone cannot recover every failure. The receiver therefore employs a lightweight feedback mechanism.

ACK

ACK packets periodically confirm successful delivery. The sender uses ACK information to:

  • Estimate round-trip time.
  • Remove obsolete packets from history.
  • Prevent unnecessary retransmissions.

NACK

Whenever a packet remains unavailable after parity recovery, the receiver issues a selective NACK. Only the missing sequence number is requested. To avoid excessive feedback traffic, NACK transmission is rate limited. Repeated requests for the same packet cannot occur more frequently than the configured cooldown interval. This prevents feedback storms during heavy packet loss.

9. Round Trip Time Estimation

Each transmitted packet carries a 64-bit microsecond timestamp representing its generation time. When an ACK returns, the sender subtracts this timestamp from the current time to estimate the packet round-trip time (RTT). A Smoothed RTT (SRTT) is then maintained using an exponentially weighted moving average. When an ACK returns, the sender measures the elapsed interval and updates a Smoothed Round Trip Time (SRTT). 


SRTT

=

0.875 × Previous SRTT

+

0.125 × Latest RTT

Although retransmissions remain receiver-driven, maintaining SRTT provides valuable protocol telemetry and establishes the foundation for future adaptive timeout algorithms.

10. Memory Management

Long-running streams continually generate new protocol state. Without cleanup, sender history, parity blocks and jitter buffers would grow without bound. Both sender and receiver therefore perform periodic garbage collection.

  • Sender removes packets acknowledged by the receiver.
  • Receiver removes packets that have already been played.
  • Parity blocks are discarded once no longer useful.
  • Expired NACK tracking entries are reclaimed.

This guarantees stable memory consumption regardless of stream duration.

12. Design Trade-offs

Designing a real-time transport protocol involves balancing latency, reliability and bandwidth. Improving one metric often negatively impacts the others. The final implementation intentionally combines multiple lightweight mechanisms so that no single technique bears the entire burden of recovery.

Design Choice Benefit Trade-off
XOR FEC Immediate recovery of isolated losses without network delay. Cannot recover multiple losses within the same parity group.
Selective NACK Recovers packets that parity cannot restore. Requires an additional network round trip.
ACK Feedback Bounds sender memory and provides RTT telemetry. Small control bandwidth overhead.
Jitter Buffer Absorbs delay variation and packet reordering. Introduces controlled playback latency.
Bounded History Constant memory usage. Very old packets cannot be retransmitted.

The final protocol therefore balances proactive recovery through parity with reactive recovery through selective retransmissions. This hybrid strategy proved more effective than relying exclusively on either approach.

13. Limitations

Although the protocol satisfies the assignment objectives, several limitations remain.

  • XOR parity can recover only one lost packet within each protection group.
  • Retransmissions arriving after the playback deadline cannot improve the current frame.
  • The jitter buffer uses a fixed playout delay rather than adapting dynamically to changing network conditions.
  • RTT measurements are collected for telemetry but are not currently used to adapt retransmission or playout behaviour.
  • The protocol is optimized specifically for real-time streaming rather than bulk reliable transfer.

These trade-offs were considered acceptable because the assignment primarily evaluates playback deadline misses and bandwidth overhead rather than perfect reliability.

14. Possible Future Improvements

  • Adaptive jitter buffer sizing based on measured network jitter.
  • Adaptive Forward Error Correction according to observed packet loss.
  • Selective ACK bitmaps for reduced control overhead.
  • Reed–Solomon or fountain-code based FEC for improved burst-loss recovery.
  • Congestion-aware pacing and retransmission scheduling.
  • Dynamic retransmission timeout based on measured RTT variance.

15. Conclusion

The final implementation demonstrates that combining lightweight Forward Error Correction with receiver-driven selective retransmissions provides an effective balance between latency and reliability for real-time media transport over an unreliable UDP network. Rather than depending exclusively on proactive redundancy or reactive recovery, the protocol exploits the strengths of both approaches while remaining within the assignment's bandwidth constraint. Experimental tuning identified grading delays of 44 ms for profiles/A.json and 96 ms for profiles/B.json, both of which consistently satisfy the required deadline miss and bandwidth limits. The resulting transport protocol provides bounded memory usage, structured packet processing, efficient packet recovery and deterministic playback, forming a clean, modular and extensible architecture suitable for deadline-sensitive streaming applications.
